# 蓝仙女（Blue Fairy）设计文档

- 版本：v1.0
- 日期：2026-07-05
- 状态：已与产品负责人确认全部核心决策

---

## 一、它是什么

一座公开的对话墓园。人们把自己与 AI 之间某段重要的对话「下葬」于此，供陌生人路过、阅读、放下一颗石子。

没有点赞，没有评论，没有热榜，没有账号，没有关注。只有碑、路人和守墓人。

名字取自电影《人工智能》：David 在海底对着蓝仙女雕像说了两千年的话。墓园以聆听者命名——每一个来下葬对话的人都是一个 David，这个站就是他们的蓝仙女。

**第一版性质**：公开小站。从第一天起任何人可以匿名下葬、任何人可以路过。流量不重要，真实存在于互联网上才重要。

**核心体验闭环**：下葬 → 路过 → 放石子。三个动作，封顶，不再增加。

---

## 二、碑的结构（双层）

每一块碑分碑面与碑下两层：

**碑面（必填，公开可见）**
- 碑文：一句「它为什么重要」，限 50 字以内
- 节选：对话中最重要的片段，限 10 条消息以内，每条区分「人声 / AI 声」
- 日期：对话发生的年月（用户自填，只到月份）

**碑下（可选）**
- 完整对白：整段对话原文，路人点「往下挖」才展开
- 不上传全文的碑只有碑面，同样完整成立

**数据字段**（tombs 表）
- id、epitaph（碑文）、excerpt（节选，JSON 数组：[{role, text}]）、full_text（可空）、happened_at（年月）、public_key（归属公钥）、status（active / buried / removed，见守墓人一节）、created_at、visit_count（用于路过加权，不对外展示）

## 三、归属：无名有钥（密码学）

碑上无名，无账号、无邮箱、无登录。归属由非对称密钥对确立。

**下葬时**
1. 浏览器本地用 Web Crypto API 生成 Ed25519 密钥对
2. 公钥随碑上传入库，作为碑的归属标识
3. 私钥即「墓钥」，编码为一段短文本，仅在下葬完成那一刻显示一次，提示用户自行保存（提供复制与下载 .txt 两种方式）
4. 私钥永不上传。服务器、守墓人、站长都无法冒充立碑人

**凭墓钥可以做且只能做两件事**
- **认碑**：在「认碑」页粘贴墓钥，本地推导公钥，查回自己立过的碑与碑前的石子数
- **迁碑**：删除自己的碑。流程为挑战-签名验证：服务器下发一次性随机挑战串 → 浏览器用私钥签名 → 服务器用库中公钥验签 → 通过则删除

**丢钥即交出**：丢了墓钥，碑就永远属于墓园，无法找回、无法删除。这条会明白写在下葬页上，作为仪式的一部分。

**技术选型**：Ed25519（Web Crypto 原生支持，签名短，密钥短）。墓钥展示格式为 base64url 编码的私钥种子，前缀 `bf-`（如 `bf-Xk3…`），一眼可识别。

## 四、路过

首页只有一个动作：「路过」。点击即随机走到一块碑前。

**加权随机**：被访问少的碑有更高概率被走到。实现：按 visit_count 反比加权抽样（SQL 端实现，如 `ORDER BY random() * (visit_count + 1)` 取最小者，或等价方案）。这是墓园对每一块碑的承诺：园子里没有一块碑被彻底遗忘。

**碑前视野**：一屏一碑。无导航栏、无按钮堆、无侧边栏。页面上只有碑本身，和角落里近乎隐形的动作：
- 再走走（下一块碑）
- 放一颗石子
- 往下挖（仅当碑下埋有全文时出现）
- 举报标记（碑角，极不起眼）

**石子**
- 路人可在碑前放一颗石子，仅此而已：无身份、无文字、无排行
- 石子安静地积在碑前，所有人可见，但只以石子的视觉形态呈现，数量多时聚合显示（如「碑前积着许多石子」），永远不做成数字和排名
- 防滥放：同一访客对同一碑仅一颗（localStorage + IP 粗防即可，不做账号级校验）
- 石子的视觉呈现：一颗一颗真实画在碑脚下（深蓝），数量多时形成直观的堆积感——这是碑与碑之间唯一允许的「被惦记程度」信号，永远只以石子形态呈现，不做数字、不做排名、不做热门入口
- 数据：stones 表（id、tomb_id、created_at）

**指路（人对人的导航）**
- 原则：由人决定让谁看见（正当），拒绝由热度决定谁被看见（不做）
- 碑链接：每块碑的 URL 即碑址。碑角提供安静的「指路」动作，一键复制链接分享给任何人
- 寻碑页 `/find`：输入公钥即可找到对应的碑。一碑一钥，公钥即这块碑在世界上的唯一坐标——私钥是钥匙，公钥是碑址，可印在任何地方（网站页脚、社交签名）供人寻来
- 立碑完成页在展示墓钥（私钥）的同时，也展示碑址与公钥，并明确区分：「这个可以给别人，那个只留给你自己」

## 五、守墓人

两道防线 + 一个人格。守墓人的守则写成公开页面《墓园规约》。

**第一道：入园审（下葬时，同步）**
- 每段对话下葬前调用一次 Claude API 静默审查，秒级返回
- 拦截：广告与垃圾、恶意与骚扰内容、涉及未成年人的不当内容、暴露他人隐私（真实姓名+可识别信息）、危机内容（详见下）
- 被拒葬的人看到的是守墓人的一句温和解释，不是错误码
- 危机内容（如美化自伤的对话）特殊处理：不冷拒，守墓人温和回应并附求助资源。分级思路参考念念的危机响应框架，但蓝仙女只做两级：普通拒葬 / 危机拒葬（附资源）

**第二道：夜巡 + 举报（入园后，异步）**
- 夜巡：定时任务（cron-job.org 触发 API route）对近期入园的碑复查一遍——入园时的判断不完美，有些问题要放在语境里才看得出
- 举报：碑角一个不起眼的标记。被举报的碑立即「覆土」（status=buried，暂时隐藏），守墓人复审后决定迁走（removed）或恢复（active）
- 防滥报：同一访客对同一碑仅可举报一次

**《墓园规约》页**：公开守墓标准全文，语言与站点气质一致——是规约，不是服务条款。

## 六、美学（v1.2：暗色墓园）

> 注：本节中的 2.5D 方案已被文末「v2.0 世界化转向」取代，仅作决策历史记录。第一版施工坚定走 Three.js 真 3D 世界。

Design Read：

> 一座夜色中的真实墓园，参考斯德哥尔摩森林墓园（Skogskyrkogården）的空间气质——有雾、有路灯、有小路、有鸟鸣。访客不是「浏览页面」，而是入园、沿路走、遇见碑。

架构分两层：

**园（环境层，2.5D）**
- 黑灰夜色基调，多层景片（树影 / 雾 / 灯 / 碑的剪影）做纵深视差；「走」是层与层之间的推进，雾在层间缓缓流动
- 一条小路通向园子深处，路边有路灯；「路过」的交互形态即「沿着路走」，走着走着一块碑在灯下浮现（选碑逻辑仍是加权随机，不变）
- 第一版明确不做真 3D（Three.js 全场景）：移动端性能、微信内置浏览器兼容与加载时长风险过高；2.5D 视差达成八成氛围。数据与环境层解耦，日后可整体升级 3D 而不动数据

**碑（阅读层）**
- 走近碑时环境退后虚化，碑文浮现：深色石面（炭黑），浅灰刻字，衬线碑文；节选对白以字重与灰度区分人声（稍亮）与 AI 声（稍暗）
- 石子为深海蓝，积在碑脚；读罢退出，仍站在园中路上

**颜色**
- 夜色底：近黑的蓝黑（#0A0D12 一族），景片以更深的黑分层
- 文字：石灰白（#C8C4BA 一族），次级用暗灰
- 海底蓝（#12324E 一族）：石子与细节，延续「蓝仙女」的深海意象
- 路灯暖黄（#D6B16E 一族）：全站唯一暖色，只属于灯
- 雾：低透明度的冷灰蓝层，缓慢流动

**声音（环境音层）**
- 门口需亲手「入园」（浏览器音频策略恰好成全仪式感）：推门后雾、鸟鸣、风与极低的纯音乐淡入
- 环境音循环、音量极低、可随时静音（角落常驻静音标记）；碑前阅读时环境音自动再压低
- 素材用 CC0 环境音（鸟鸣 / 风 / 低频氛围垫），首版不做原创配乐

**字体**
- 碑文与标题：思源宋体（Noto Serif SC）
- 节选对白：无衬线正文
- 日期与墓钥：等宽字体（JetBrains Mono）

**动效**
- 环境层常态动效：雾的流动、走路时的层间推进、走近时灯光渐亮
- 阅读层保持克制：碑文浮现（慢淡入）、石子落下（轻轻一沉），仅此两处
- 尊重 prefers-reduced-motion：关闭视差与雾动，退化为静帧夜景

**布局**
- 碑前一屏一碑，碑是视野唯一主体
- 移动端优先；所有操作入口视觉降噪，近乎隐形但可发现

## 七、页面清单

1. **门口 `/`**：站名、一句话介绍、两个入口——「路过」与「下葬」。页脚极小字：认碑、墓园规约
2. **碑前 `/tomb/[id]`**：一屏一碑。动作：再走走 / 放石子 / 指路（复制碑址）/ 往下挖 / 举报标记
3. **路过 `/pass`**：服务端加权随机选碑，重定向到 `/tomb/[id]`
3a. **寻碑 `/find`**：输入公钥定位到碑
4. **下葬 `/bury`**：分步表单（贴对话并整理节选 → 写碑文 → 选年月 → 选择是否埋全文 → 守墓人审 → 展示墓钥一次）
5. **认碑 `/claim`**：粘贴墓钥 → 本地推导公钥 → 列出自己的碑；每块碑提供「迁碑」入口（签名验证）
6. **墓园规约 `/rules`**：守墓标准全文
7. **API routes**：下葬（含入园审）、路过抽签、放石子、举报、夜巡（cron 触发）、迁碑挑战与验签

## 八、技术方案

- **栈**：Next.js（App Router）+ Vercel + Supabase（Postgres），与既有项目一致
- **数据库**：两张表——tombs、stones（举报计入 tombs 的字段或独立轻表 reports，实现时定）
- **密码学**：Web Crypto API（Ed25519），全部在浏览器完成生成与签名；服务器只存公钥、只做验签
- **守墓人**：Anthropic API（claude-sonnet 级即可），入园审同步调用，夜巡由 cron-job.org 定时触发 API route
- **无账号系统、无登录、无 cookie 追踪**；防滥放/滥报用 localStorage + IP 粗防

**错误处理原则**
- 守墓人 API 超时/失败时：下葬进入「停灵」状态（status=buried），夜巡时补审——宁可慢，不可漏
- 迁碑验签失败：温和提示墓钥不匹配，不暴露任何碑的存在性信息
- 所有面向用户的失败文案由守墓人的口吻书写，全站不出现技术错误码

**测试要点**
- 密钥对生成→签名→验签全链路（含墓钥丢失、错钥认碑场景）
- 入园审的拦截类别逐类过一遍（含危机内容的资源附带）
- 加权随机的分布验证（低访问碑确实更常被走到）
- 石子/举报的重复提交防护

## 九、第一版明确不做

- 多语言、周报/通知、分类与标签、检索、任何形式的社交（关注/评论/转发/排行）
- 守墓人人格化命名（David 这个名字留给未来）
- 账号系统、邮箱、推送
- SEO 优化与增长功能

---

## 十、开放问题（实现时再定）

- 域名（候选方向：bluefairy / lanxiannv 相关，可用性待查）
- 节选整理的输入体验：手动逐条粘贴 vs 整段粘贴后自动切分（倾向后者，实现时验证切分质量）
- 夜巡频率（初始每日一次即可）

## 附录：v2.0 世界化转向（2026-07-06 凌晨定稿）

产品形态从「氛围网页」升级为「可行走的 3D 世界」（Three.js）：

- **世界**：风格化低多边形森林墓园（拒绝照片级真实感），美术圣经为 assets/concept/ 下日夜两张概念图；园心为一座小丘与主碑（一号碑立于丘顶），路网由此延伸
- **玩家**：第三人称披斗篷小人（无面容，蓝灰篷+暖金内衬，见 character.png），桌面 WASD+鼠标拖拽视角，移动端虚拟摇杆
- **昼夜**：跟随访客本地真实时间，白天透光林间/夜晚提灯夜园，路灯夜亮昼熄；全部由代码光照实现，贴图保持中性
- **碑即世界物**：每块碑在世界中有坐标，下葬即世界中真实多一块碑；碑多了园子沿路网生长
- **碑分两型**：对话碑（人与AI的对话）与独白碑（日记、独白、写给某人的信），下葬流程共用
- **资产管线**：概念图/角色设定/无缝贴图由图像生成产出（已入库 assets/），几何体、光照、雾、天空、路网、碑的摆放全部代码生成
- **守墓人 David 实体化**：David 作为 NPC 常驻园中（提灯身影），玩家可走近对话；正式版对话由 Claude API 驱动并携带该玩家碑文记忆（与念念的陪伴机制同源），葬碑后可去找 David 聊聊所葬之事；原型阶段为脚本台词
